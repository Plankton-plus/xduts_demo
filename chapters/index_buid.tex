\chapter{索引构建}
本文从整体上采用滑动窗口机制，也就是索引会有一个窗口生命周期管理，在生成一个新区块时，旧块数据会在索引中删除，新块数据会加入索引。
索引始终维护的是最近窗口大小的数据状态，链尾区块记录的也就是当前索引的最新快照。
\section{密文索引构建}
\subsection{对象注册}

本文采用的密码学累加器对输入全集大小敏感，其证明生成复杂度为 $O(N_1 \cdot N_2)$，其中 $N_1, N_2$ 分别为输入集 $X_1, X_2$ 的大小。相较于其他累加器\cite{xu2019vchain}，该方案虽支持交、并、差等嵌套集合操作，但需以更大的证明为代价，且公钥大小与输入集合元素的全集大小呈二次关系。若直接将滑动窗口中的数据对象通过加密散列编码为 $256$ 位整数，公钥规模将达 $(2^{256})^2 = 2^{512}$，难以接受。

为此，我们提出对象注册索引 ObjReg，为最近 $2^K-1$ 个区块内的数据对象分配有限 ID，并为后续属性与范围索引提供统一标识。通过设定最大 ID（记为 $MaxID$），将参与累加器运算的输入集大小限制在 $MaxID$ 内，从而控制公钥规模。例如，在实验中将 $MaxID$ 设为 $2^{12}$，公钥大小即为 $(MaxID)^2 = 2^{24}$。该机制同时保证任意连续 $2^K-1$ 个区块内的数据对象均具有唯一 ID，而本文涉及的集合操作仅在这一窗口范围内进行，因此所有相关对象均可确保 ID 不重复。

ObjReg 是一棵扇出为 $f$ 的完全平衡 Merkle 树，其根哈希写入区块头，用于承诺以该区块结束的窗口状态，同时计入区块体以供一致性检查。对象注册时，矿工为其分配一个随计数器递增并模 $MaxID$ 的 ID。由于树结构固定，$ID_i$ 可视为以 $f$ 为基数的整数：
\begin{equation}
ID_i = \sum_{j=0}^{h-1} d_j \cdot f^j \quad \text{其中 } d_j \in [0, f-1]
\end{equation}
每位 $d_j$ 对应第 $j$ 层的子节点选择，Merkle 树子节点从 $1$ 编号时，路径取 $d_j+1$。如图 \ref{fig:objreg} 所示，设扇出 $f=3$，对象 $o_6$ 的 ID 为 $6$，其三进制表示为 $020$，对应树路径为 $(1, 3, 1)$。

叶节点存储对应对象标识符位置的加密承诺及其哈希，内部节点仅存储子节点哈希的聚合值。每个区块维护自身对应的 ObjReg 根，系统中不存在一个横跨全链的单一全局根，而是形成一条随区块演化的状态序列。后续索引不再引用原始对象，而是统一引用这些经过ObjReg分配的对象标识符。

\section{关键字索引AttDigest}
起初我们设想AttDigest继续按照ObjReg的ID位置插入，那么其本质就是带属性摘要的ObjReg镜像树。这样更新简单，但剪枝效果很弱，因为树的空间邻近性来自ID，而不是来自属性相似性，所以我们需要找到某种组织方式使得AttDigest按相似语义聚集。回忆先前设计，系统初始化时就已经把关键词压缩成了稳定稠密的词表编号，所以关键字经过PRF在词表中对应的$kw_{id}$就是此时AttDigest的索引键。语义上搜索标签token等于词表编号$kw_{id}$。
自然而然地，关键字索引AttDigest设计为和ObjReg一样的固定扇出、按$kw_{id}$唯一路径定位的认证倒排树。每个叶子唯一对应一个$kw_{id}$，因此AttDigest的路径由$kw_{id}$唯一确定，而不再依赖对象分布或插入顺序。下面给出每个索引节点定义：

\begin{definition}
(AttDigest Leaf Node) 设叶节点 \(n\) 对应关键字 \(k\)，其字段定义如下：
\begin{itemize}
    \item \(kw_{id}\) = 当前叶对应的关键词编号
    \item \(S_{n}\) = \(k\) 对应的对象 ID 集合
    \item \(acc_{n} = acc(S_n)\)
    \item \(cnt_n = |S_n|\)
    \item \(h_n = H(kw_{id} \parallel acc_n \parallel cnt_n)\)
\end{itemize}
\end{definition}

\begin{definition}
(AttDigest Non-Leaf Node) 定义非叶节点的子节点为 \(c_1, c_2, \dots, c_F\)，其字段定义如下：
\begin{itemize}
    \item \(acc_n = acc(K_n)\)，其中 \(K_n\) 是当前子树下非空关键字编号集合
    \item \(h_n = H(acc_n \parallel h_{c_1} \parallel h_{c_2} \parallel \dots \parallel h_{c_F})\)（如果 \(n\) 不是根节点）
    \item \(h_n = H(acc_n \parallel h_{c_1} \parallel h_{c_2} \parallel \dots \parallel h_{c_F} \parallel height)\)（如果 \(n\) 是根节点）
\end{itemize}
\end{definition}

对于叶节点n，若当前窗口内没有任何对象包含关键字k,则令$S_k = \emptyset$,并计算相应空集合累加器摘要与空叶哈希。请注意叶节点存储的是$kw_id$与对象标识符集合的认证摘要，而不是明文关键字或者32字节搜索标签本身。这样既压缩了节点存储开销，又避免了任意字符串域直接映射到小整数域所带来的碰撞问题，因为系统中的整数编号不是由简单压缩得到，而是由共享词表对全体关键字进行一致编号后产生。

对于内部节点n，实际上其所覆盖的关键字编号区间$K_n$是由树结构隐式决定的，但我们仍存储其累加器摘要$acc_n$是为了批量查询优化与空子树快速判断，它并不是单关键字查询正确性的必要条件。单关键字查询的正确性已经可以由$kw_{id}$唯一路径定位的叶子认证路径保证；该字段的存在只是为了在将来支持多关键字联合调度、批零空子树等工程优化时，提供附加的认证辅助。

如算法表示：在窗口末端 $b_h$ 处，系统首先枚举该窗口内的所有对象标识符。对于每个对象，获取其对应的关键字编号集合，并将该对象的标识符分别添加到每个关键字所对应的倒排集合中。全部对象处理完毕后，即可得到当前窗口下完整的倒排集合族。随后，对每个关键字的倒排集合计算累加器摘要，形成树的叶节点，再自底向上递归计算内部节点的累加器摘要与哈希值，最终得到当前窗口的 attdigest 根哈希。该根哈希被写入区块头，作为窗口 $b_h$ 对关键字索引状态的唯一承诺。

\section{密文索引更新}

由于系统采用滑动窗口模型，当新区块$b_{h+1}$到达时，索引更新不是简单插入，而是一个先删除过期对象，再插入新对象的状态转移过程。

设当前窗口对应区间为$[h-w+1,h]$。当新区块$b_{h+1}$到达时，若窗口已满，则原窗口左端块$b_{h-w+1}$中的对象全部过期，需要从ObjReg、AttDigest和RangeDigest中删除;随后，再将新区块中的对象注册进ObjReg，并加入两类索引。这样，系统状态从窗口$[h-w+1,h]$平滑转移到窗口$[h-w+2,h+1]$。

\subsection{ObjReg更新}
对每个过期对象，系统依据其旧的对象标识符，将对应叶节点恢复为空值，并沿路径认证自底向上重算到根；对于每个新对象，系统分配新的对象标识符或重复利用已释放的标识符，并填充到相应叶位置，再重算到根，从而得到新的ObjReg根。
\subsection{AttDigest更新}

系统为高效维护关键字倒排索引，额外维护了一个反向映射，记录每个对象标识符$id_{o_i}$所对应的全部关键字编号。这样，当对象过期时，系统无需重新解析其明文字段，可直接通过该映射定位到受影响的叶节点。

对于每个过期对象，系统首先获取其关联的关键字编号集合。随后，对集合中的每个关键字，从对应的倒排集合中移除该对象的标识符，并重新计算受影响叶节点的累加器摘要与计数信息，再沿关键字路径自底向上更新祖先节点的摘要与哈希值。对于新区块中的新对象，则执行相反流程：读取其关键字编号集合，将对象标识符依次插入相应倒排集合，并沿路径完成更新。

因此，AttDigest的更新代价不取决于整棵树的大小，而只取决于过期与新增对象的关键字数量以及树高。由于节点键由$kw_id$唯一确定，更新过程中不会发生树结构重排或者节点语义重写，从而保证该索引在滑动窗口模型下是可删除且可增量维护的。
