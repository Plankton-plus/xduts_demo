方案设计：

ObjReg是：
构建会维护一个模maxID 的递增计数器，为新对象分配唯一ID并且$$ID_i\in[0,maxID-1]$$以确保在最近$$2^k-1$$个区块内，该ID不重复。由于ObjReg是固定扇出为$$f$$的树，所以可将$$ID_i$$视为一个以$$f$$为基数的整数:
$$ID=\sum_{j=0}^{h-1}d_j\cdot F^j\quad\mathrm{where~}d_j\in[0,F-1]$$
每一位$$d_j$$对应ObjReg中第$$j$$层的子节点选择。由于Merkle树的子节点通常从1开始编号，因此树路径为$$tree\_index_{j} = d_j+1$$。如下图所示，假设ObjReg扇出$$f = 3$$，待插入对象$$o_6$$的ID为6，则$$6 = 0\times3^0+2\times3^1+0\times3^2$$，其三进制表示为020。因此其在ObjReg中的树路径为$$(1st,3rd,1st)$$,即第1，第2和第1个子节点。
在生成新区块时，我们除了插入新对象，还必须显示地将过期（超出滑动窗口）但尚未被覆盖的旧对象位置重置


ObjReg需要提取出数学原型，当前设计中所有对象都存储到叶子节点上了，大量的非叶节点仅仅用于哈希认证。这个索引设计效率太低。
并且，移除元素总是从最左侧元素移除，需要从底往上重新哈希。
原因在于假设位长为8，值为2，其二进制是00000010。我们依次找路径为1，1，1，1，1，1，2，1。如果我们认为前缀0无效，则其路径为2，1。倒过来是1，2.
现在对比：我们原来方案是认为二进制前导0都路径有效，所有元素存储在叶子节点。非叶节点仅仅计算其子节点哈希，然后层层向上计算出根哈希。

\section{关键字索引优化}
关键字索引构建

优化方式：
原有方式是按照ID决定插入位置，所以最终决定的树结构会和objreg一模一样，甚至可以合并为一个树。只不过我们为了设计更清晰，所以独立出一个树。
但是这样不能按照属性相似聚集在一起，以便给出更少的不相交证明。所以有什么更好的插入方式，更好的树组织方式能够减少给出不相交证明，聚合相似节点在一起。
底层叶子节点是一个对象及其属性。当插入一个新节点时，我们先将已经过期的元素删除，从上至下计算相似度决定往下走哪个分支，直至叶子节点。
父节点会存储子节点的所有属性，分为子节点相交属性，还有除相交之外属性（附加子节点映射）。举例来说，子节点属性是o1:a,b,c。o2:a,b,d。那么父节点就是:a,b,(c->o1),(d->o2).
查询时，先与所有节点验证相交，如果都没有，则折叠成哈希。如果都有，则将该非叶节点包括的对象ID全部纳入结果，如果有一部分，则往下继续深入。
但这有个问题，就是叠加了滑动窗口后，对旧有数据删除很复杂，以及哈希更新也很复杂。方案设计还要考虑acc值更新，验证数据生成。
这样组织有点像属性集合上的BTree树，但不知道理论和实际上是否可以实现。

如果AttDigest继续按照ObjReg的ID位置插入，那么其本质就是带属性摘要的ObjReg镜像树。这样更新简单，但剪枝效果很弱，因为树的空间邻近性来自ID，而不是来自属性相似性。
所以我们需要找到某种组织方式更贴合attdigest的“按查询语义聚集”
如果按照这样的优化方式，首先父节点语义不是局部可增量更新。比如有两个孩子s1 = {a,b},s2={a,c}，父节点公共部分是{a}.将s2删除后，父节点公共部分会从{a}变为{a,b},反而
增加了属性。这说明父节点需要重新扫描所有剩余孩子才知道公共属性和参与映射是什么，进而会导致路径上每个祖先的属性分解都需要重写。

最自然的方式是按照倒排索引组织，attdigest维护的是tokenid -> objectIDSet。 objectIDSet表示当前滑动窗口内，所有包含该tokenid的对象集合。
每个叶子是一个token桶，叶子里存这个token对应的对象ID集合。

叶节点
token:
s_token
acc_id(tok)
h_leaf = H(tok||acc_id(tok)||meta)

首先这个token指的是将关键字经过PRF后的哈希值再继续在词表中查询的id,在代码中类型是NonZeroU16。也就是token语义上等于tokenid。
其次还需要补充一个acc_token_id.
因为当令牌中的token和当前节点取交集时，如果不匹配，还需要给出（token_id,keyword_trapder_id)->(空集，证明)，以证明该节点确实不相交。
内部节点组织成这个：一种是Sparse Merkle Trie / Patricia Trie，键为 H(tok)；没有看懂。

滑动窗口删除逻辑应该和你的类似。假设有token集合{t1，t2，....tm)。对每个ti删除id(o),重算acc。沿路径重建哈希。



优化方式:按照相似对象聚集
稳定签名桶 + 桶内倒排/小树

也就是先给每个对象计算一个固定签名，例如：

MinHash 签名
SimHash 签名
取若干主导 token 的哈希前缀

然后用签名前缀决定对象落在哪个桶。

这样相似对象大概率落在相近桶中，但路径是确定的、可复现的、可删除的。
这比“根据当前树状态动态算最相似分支”稳定得多。

细节说明


查询陷门会给出kwid，如果关键字不在表内，那么结果必为空，无需上链证明。如果查询关键字在表内，那么服务端只需要返回kwid对应叶子，若叶子为空，则则证明当前窗口
无匹配对象，若posting非空，就返回posting摘要与后续结合证明。也就不需要额外证明“别的节点都不想交”，因为别的节点本来就不是这个键的位置，他们已经被Merkle路径上的兄弟哈希认证掉了。

叶节点定义字段为：
kwid:当前叶对应的关键词编号
S_kwid:kwid对应的对象ID集合
acc_obj(kwid):对S_{kwid}的对象ID集合累加器摘要
cnt(kwid):|S_kwid|，方便区分空叶和非空叶
h_leaf = H(kwid||acc_obj(kwid)||cnt(kwid)||meta)
meta可以放版本号等等其它元信息

内部节点定义为：
range：该节点覆盖的kwid区间，由树结构隐式确定
h_children(n)：子节点哈希聚合
h_n = H(h_children||meta)

其实这里可以暂时不存acc，因为按照关键字组织的包牌索引，布尔集合证明是在查询到的多个叶子posting集之间做的，而不是在内部节点做

查询过程如下：
用户输入关键字w：可在共享词表中查得kwid
情况1：kwid不存在，词表级别空，词表三方共享，无需AttDigest证明
情况2：kwid不存在
    sp在attdigest按kwid唯一路径定位到叶子，返回叶子内容，以及Merkle认证路径。
    用户验证：叶子确实对应kwid，路径重建根一致。若cnt为空，则结果为空，若cnt>0,则结果对象ID集由acc_obj承诺。这里如果漏查，会和根节点不一致。
布尔查询：
先本地得到kwid_j,然后SP返回叶子leaf(kwid_j)的acc_obj(kw_id)和Merkle Proof。再对这些posting集执行交并差，并返回集合运算证明。

滑动窗口更新逻辑：设过期对象o的关键词编号集合为K(o) = {t_1,t_2,...t_m},则对每个t_i,从S_{t_i}中扇出id(o),重新计算acc_obj,若集合为空，则把cnt置0，
自底向上重算到根。

这里必须保存oid到vec<kwid>的反向更新信息，因为当对象过期时，必须知道它曾经影响了哪些的posting叶子。需要在块内容里额外记录每个对象的kw_ids.
